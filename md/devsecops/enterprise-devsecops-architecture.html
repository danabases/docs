§ 01 / Principles

Four commitments that shape every decision.

Before any tool is chosen, these commitments decide what good looks like. They are the reason we build templates instead of mandates, measure instead of gate, and move in small stages instead of large leaps.

I.

Platform
not team.

Everything we adopt must be consumable by orgs that don't report to us. Every pattern has a shareable form — a template, a reusable workflow, a documented standard.

II.

Small steps,
staged rollout.

New territory for most participants. We pilot in low-risk repos, measure, then expand. The COE agenda moves one or two artifacts forward per sprint — no big-bang.

III.

The right thing,
made easy.

Resistance to standards is usually resistance to friction. Templates, workflows, and instruction files exist to reduce effort — not to enforce through pain.

IV.

Visibility over
gatekeeping.

Zero Trust means continuous verification, not continuous blocking. Measurable posture, attestation, evidence — these compound. Manual approval gates do not.

§ 04 / Flow

How a policy travels.

A single standard moves through four distinct states on its way from drafting to daily developer experience. Each stage is someone's responsibility; each produces a specific artifact that survives the stage.

01
Draft

Decision proposed in COE

An artifact enters the backlog. Either a draft ADR, a prototype repo, or a sample workflow. Opinions without artifacts get parked.

02
Codify

ADR + reference repo

Decision recorded, numbered, immutable. A reference implementation lives in the shared platform repo, versioned and tagged.

03
Publish

Template + reusable workflow

Turned into a consumable primitive. Repo template, starter workflow, Copilot instruction file, MCP config. Discoverable from the docs site.

04
Adopt

Measured in posture

Adoption surfaces in the compliance dashboard. Evidence flows to Enterprise Security automatically. Audit cycle takes days, not weeks.

The stack earns its keep through context.

A developer opening a repo hits every layer in order. The profile loads the right extensions. The workspace tells Copilot which repos to consider together. Instruction files shape what Copilot writes. Knowledge bases ground the answers. MCP servers let Chat reach into real systems — a live SQL schema, an Azure policy assignment, a PR review thread.

Most of the value of Copilot Enterprise is in features people don't know about. Instruction files, knowledge bases, MCP — these turn an autocomplete into a teammate who knows how we work.

No layer requires the ones above it. A team can adopt profiles without workspaces, workspaces without MCP, MCP without knowledge bases. We publish the stack so each team can start where it makes sense for them.

§ 06 / Rollout

Five stages, none skipped.

Each stage is a COE agenda item with an artifact. We don't start a stage until the prior one is stable. Stages are not dates — they're conditions.

01
Foundations

Mostly invisible to developers.

SSO and SCIM enforced. Audit log streaming to the enterprise SIEM. Enterprise-level secret scanning with push protection. Dependabot on by default. Low-risk, high-leverage, and the prerequisite for everything that follows.

SSO · SCIM
Audit log streaming
Secret scan + push protection
Dependabot defaults
02
Baseline CI

A floor for every pipeline.

Required reusable workflow for security scans on a pilot set of repos. CodeQL on all supported languages. SBOM generation per build. Measured by coverage percentage across tagged repos.

Required reusable workflows
CodeQL coverage
SBOM generation
Coverage dashboard
03
Provenance

Everything built is attributable.

OIDC federation for all cloud deployments — no long-lived credentials in GitHub Secrets. Signed commits on any repo with compliance scope. Build provenance attestations generated on release. SLSA level framing applied where relevant.

OIDC cloud auth
Signed commits
Build attestations
SLSA posture
04
Differentiated

Policy targets property, not name.

Rulesets targeting custom properties replace hand-edited branch protection. Stricter controls on restricted and CJIS-scoped repos. Environment protection rules on production deployments with required approvers.

Property-based rulesets
CJIS / restricted controls
Environment protections
Required approvers
05
Continuous

Evidence flows, not gathered.

Dashboards aggregating posture per repo. Automated audit evidence bundles. Compliance scope maps to a defined control set automatically. Enterprise Security consumes telemetry instead of requesting it. Audit cycle becomes a review, not an archaeology dig.

Automated evidence
Control-set mapping

§ 07 / Anchors

The standards we speak.

Security architecture choices reference established frameworks rather than inventing new ones. Auditors, customer orgs, and Enterprise Security all recognize this vocabulary. Deviations are recorded as ADRs.

§

Development Framework
The baseline for how CI/CD security controls are structured. Maps directly to our reusable workflows.
§
NIST · SP 800-207
Zero Trust
Architecture
Applied to identity, device, workload, and data. Shapes our OIDC, attestation, and continuous verification posture.
§

Control Catalog
The lingua franca when customer orgs ask which control a workflow maps to.
§
CJIS Security Policy

Information Services

§

§
OWASP · SAMM / ASVS

for application security
Structures how we measure appsec maturity across orgs and define the verification bar per tier.
§ 08 / COE

A body that ships, not one that meets.

The DevSecOps Center of Excellence fails when it produces slides without code and succeeds when every session moves at least one artifact forward. Its output — not its cadence — is the measure.

The COE is an artifact-producing body. A proposal must carry a draft ADR or a prototype repo. Opinions without artifacts get parked.
I.
Architecture Decision Records numbered, immutable
II.
Standards documents checklists, not essays
III.
Reference implementations real, running repos
IV.
Reusable workflows & templates the path of least resistance
V.
Adoption measurement posture over promise
